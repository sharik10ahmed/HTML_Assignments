Slide 05 · Case Flow — Part I

The Case Lifecycle:
Initiation to Assignment in Four Steps

A deterministic state machine — every case traverses four validated checkpoints before a live lawyer workspace is activated. No checkpoint can be bypassed programmatically.

Initiation
CASE_CREATED
Client selects a verified lawyer from the marketplace directory. POST /api/client/cases fires — a CaseRequest record is created with a generated custom_id (e.g. ci:12345) linking client_id and lawyer_id.
Document Upload
DOCUMENT_VERIFICATION
Client uploads required legal documents. Each upload creates a CaseDocument row in PostgreSQL and a corresponding CaseDocumentVerification record with status PENDING, automatically feeding the admin review queue.
Admin Verification Gate
DOCUMENTS_VERIFIED
Admin reviews and approves each document individually. An aggregate check fires after every approval — when all documents reach VERIFIED, the CaseRequest auto-flips to DOCUMENTS_VERIFIED. No manual trigger needed.
Lawyer Assignment
LAWYER_ASSIGNED
The client_id ↔ lawyer_id relationship is confirmed. Live case workspace activated, case chat opened, both parties receive a WebSocket notification instantaneously.

Slide 11 · Ledger & Payments

The Payments Engine:
Razorpay-Verified, Signature-Protected, Audit-Complete

Zero-trust financial architecture. Every payment success callback is cryptographically validated via HMAC-SHA256 before a single database record is mutated. Spoofed callbacks are silently rejected.

STAGE 1 · ORDER INITIATION
Order Created
Client clicks "Pay Now." Platform immediately creates a Payment DB row with status PENDING and invoice inv:12345 — before the payment gateway even opens.
POST /payments/create_payment
→ invoice: inv:12345 · status: PENDING
STAGE 2 · CHECKOUT & CAPTURE
Razorpay Checkout
PCI-compliant Razorpay modal launches. Client transacts via Card / UPI / Net Banking. Payment captured by Razorpay servers — razorpay_payment_id returned in success callback.
Returns: razorpay_payment_id + razorpay_order_id
STAGE 3 · VERIFICATION & DISPATCH
Zero-Trust Signature Gate
Backend computes HMAC-SHA256(order_id + payment_id, secret) and validates against Razorpay's signature. If valid — DB → SUCCEEDED, WebSocket fires, NATS dispatches receipt.
POST /{payment_id}/verify → status: SUCCEEDED
🔐
HMAC-SHA256 Signature Gate — Zero-Trust Financial Architecture
Client Pays → Razorpay Captures → Signature Verified → DB → SUCCEEDED → WebSocket Alert → NATS Receipt Email → inv:12345 delivered
